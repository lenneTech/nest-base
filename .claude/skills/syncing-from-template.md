# Syncing From Template

How to pull `src/core/` updates from the upstream template into a
project that consumes it. The hard guarantee: **`src/modules/` is
never touched**.

## When you'd run this

- The template has a new release (subscribe to the template repo's
  tags)
- Before starting a larger feature so you don't fight stale
  infrastructure
- After any reported security issue against `src/core/`

## The command

```bash
bun run sync:from-template
```

The runner:

1. Clones the template repo to a temp dir
2. Reads its `src/core/**` snapshot
3. Calls `planSyncFromTemplate({ templateCore, local })` (the pure
   planner)
4. Applies the resulting operations through `node:fs/promises`
5. Prints a per-bucket summary

## What gets touched

| Bucket     | Meaning                             | Action                        |
| ---------- | ----------------------------------- | ----------------------------- |
| **create** | File only in template               | Written into your `src/core/` |
| **update** | Both have it, content drifted       | Overwrites your local copy    |
| **skip**   | Both have it, content matches       | No write                      |
| **delete** | File only in your local `src/core/` | Removed from your tree        |

Output is byte-deterministic — re-running on the same trees gives the
same plan, so commits stay clean.

## What does NOT get touched

- `src/modules/` (always)
- `src/shared/` (always)
- `tests/` (always)
- `prisma/` (always)
- `package.json`, `tsconfig.json`, anything outside `src/core/`

The planner refuses any _upstream_ path outside `src/core/` with
`ProtectedPathTouchedError`. A misconfigured runner (or a maliciously-
crafted template snapshot) cannot smuggle writes into `src/modules/`
this way.

## After the sync

A `sync:from-template` may have changed Prisma feature schemas. Re-run
the schema concatenation and migrations:

```bash
bun run prepare:schema
bunx prisma migrate deploy
```

Then your usual quality gates:

```bash
bun run lint
bun run test:e2e
bun run build
```

If anything breaks, the diff is in `src/core/` — `git diff` and decide
whether to:

1. Adopt the upstream change (commit as-is)
2. Override locally (revert the unwanted hunk; document the
   divergence in `OPEN_QUESTIONS.md` so the next sync sees it)
3. Send a PR upstream changing the template (see
   `core-contribution-guide.md`)

## Reviewing the diff

Use `git diff src/core/` after the sync. The deterministic output
keeps the diff small enough to scan; what was generated by the planner
is exactly what's in the diff.

For larger drifts, the runner prints a per-bucket count:

```
[sync] create: 3 file(s)
[sync] update: 7 file(s)
[sync] skip:   42 file(s)
[sync] delete: 1 file(s)
```

## Long-lived divergences

If your project genuinely needs `src/core/` to deviate from the
template (rare — the right move is usually a PR upstream), record it
in `OPEN_QUESTIONS.md` with a clear heading:

```markdown
### project-local-divergence: src/core/<file>

- **Reason**: <why we deviate>
- **Hunks affected**: <function names>
- **Reconsider when**: <condition that would let us drop the
  divergence>
```

The sync planner doesn't read this file — it's a human checkpoint so
the next sync reviewer knows the override is intentional.

## Committing the sync

The convention is one commit per sync, with the bucket counts in the
message:

```bash
git add -A
git commit -m "chore(sync): pull from template@<sha-or-tag>" -m "$(cat <<'EOF'
- create: 3
- update: 7
- delete: 1
- skip:   42

Notable changes: <one-line summary per non-trivial update>
EOF
)"
```

If a migration was generated, commit it in the same commit as the
schema change.

## Don't

- **Don't edit `src/core/` directly** unless you've decided to live
  with the override. The next sync will overwrite you.
- **Don't bypass the planner** by manually copy-pasting from the
  template. The defense-in-depth on `src/modules/` lives in the
  planner; bypassing it puts the project at risk.
- **Don't skip the post-sync gates.** Failed gates after a sync are
  the whole point of having gates.
