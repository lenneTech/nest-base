## Methodology

The planner walks every `*.controller.ts` and `*.module.ts` under `src/`,
parses each `@Get/@Post/@Put/@Patch/@Delete/@All` decorator, collects
the surrounding `@Can(...)` and `@Public(...)` decorators (above and
below the http method), resolves the full path (controller base +
handler argument), and classifies:

- **`gated`** — handler carries `@Can(action, subject)`. The
  `CanGuard` evaluates the active CASL ability at request time
  ([architecture.md → Permission model](../architecture.md#permission-model)).
- **`public-by-design`** — handler carries `@Public("<reason>")`
  *or* the resolved path matches one of the runtime public allowlists
  (`PUBLIC_PREFIXES` / `PUBLIC_EXACT` in
  [`src/core/auth/jwt-middleware.ts`](../../src/core/auth/jwt-middleware.ts),
  `EXEMPT_*` in [`src/core/multi-tenancy/tenant-guard.ts`](../../src/core/multi-tenancy/tenant-guard.ts)).
  Includes `/dev/*` and `/admin/*` — both 404 outside
  `NODE_ENV=development` via `assertDev()`, so they cannot leak in a
  production build.
- **`ungated-bug`** — neither. Goes through every layer of the
  permission stack as ANONYMOUS, relying solely on Postgres RLS as the
  last line of defence. **Defense-in-depth lives or dies on three
  layers; ungated is one bad query away from CVE-class data leakage.**

### Rationale for `@Public()` choices

The two `@Public()` routes are bootstrap-only. Both require an
authenticated `req.user` (Better-Auth session middleware), and both
return data scoped to that authenticated identity — but neither can
sensibly carry `@Can("..", "Tenant")` because:

- `Tenant` is intentionally absent from the default Member-role grants
  ([`member-role-rules.ts`](../../src/core/permissions/member-role-rules.ts))
  — it is framework-admin.
- A fresh user who has just signed up and has zero memberships needs
  these routes to bootstrap. Gating with a `Tenant` ability that no
  fresh user holds would create an unbreakable chicken-and-egg
  bootstrap.

The `@Public(reason)` decorator surfaces the consent at the decoration
site so an auditor scanning these files sees a deliberate decision,
not a missed gate.

### Per-user vs per-tenant scoping

The audit added a second knob to the default Member-role rule planner
in [`member-role-rules.ts`](../../src/core/permissions/member-role-rules.ts):

```ts
DEFAULT_MEMBER_RESOURCES         // tenantId = $CURRENT_TENANT
DEFAULT_MEMBER_PER_USER_RESOURCES // userId   = $CURRENT_USER
```

The per-user list seeds `ApiKey` and `Session` so a regular tenant
member sees only their own keys / sessions, regardless of which
tenant they're operating in. CASL evaluates BOTH lists when building
the request-scoped ability, and the Postgres RLS layer keeps the last
line of defence regardless of the planner's output.

## How to keep this clean

1. **CI gate** — `tests/stories/route-gating-audit.story.test.ts`
   fails the suite when a route lands without a marker. Re-run
   `bun run test:e2e tests/stories/route-gating-audit.story.test.ts`
   locally to verify before pushing.
2. **Adding a new route** — pick `@Can(action, subject)` for anything
   that touches user-owned data. If the route is genuinely
   anonymous-by-design (health probe, callback endpoint your auth
   plugin owns, public OAS catalogue), use `@Public("<reason>")` with
   the why at the decoration site.
3. **Adding a new public allowlist prefix** — extend
   `DEFAULT_PUBLIC_PREFIXES` in
   [`src/core/permissions/route-audit-planner.ts`](../../src/core/permissions/route-audit-planner.ts)
   AND `PUBLIC_PREFIXES` in
   [`src/core/auth/jwt-middleware.ts`](../../src/core/auth/jwt-middleware.ts)
   in the same commit so the audit and the runtime stay in sync.
4. **Re-generate this report** — when you add or remove gated routes,
   run the planner from the repo root and replace this table:

   ```bash
   bun run -e 'import { auditControllerRoutes } from "./src/core/permissions/route-audit-planner.ts"; …'
   ```

## Out of scope

- `src/modules/**` — project-owned, not template-shipped. The audit
  walks them anyway (see `ExampleController` / `UserProfileController`
  rows above) so a project that imports the gate gets the same
  build-time check, but the gate itself is enforced by the template's
  CI.
- Permission-Tester UI banner (Phase 4 of #47) — deferred. The CI
  gate already prevents drift; surfacing the count in the dev-portal
  page is a follow-up convenience.
